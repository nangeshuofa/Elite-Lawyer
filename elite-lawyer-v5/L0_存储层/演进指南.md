# 演进指南 — 如何持续演进本技能

> 本文件指导维护者进行三类演进操作：**升级版本**、**迁移数据**、**扩展能力**。
> 核心原则：每次演进都必须通过 `架构约束.md` 的三性平衡检查（稳定性 > 可维护性 > 可扩展性）。

---

## 一、版本升级流程

```
Step 1 — 确定版本号（语义化：主.次.修）
  - 主版本（V6→V7）：架构层变更（如新增/删除层级、改变依赖方向）
  - 次版本（V6.0→V6.1）：新增能力（新模块/新领域/新机制），向后兼容
  - 修版本（V6.0.1）：内容修正（错别字、知识更新、提示优化）
Step 2 — 更新 SKILL.md 头部版本号 + 版本历史段
Step 3 — 如涉及 schema 变更 → 见第二节数据迁移
Step 4 — 更新 README.md 版本说明
Step 5 — 重新打包 .skill 文件
Step 6 — 提交 + 推送 + 打 Release 标签（如 v6.0）
```

---

## 二、数据迁移流程（schema 变更时）

```
Step 1 — 在 存储契约.md「Schema 版本记录」表追加新版本行
Step 2 — 定义迁移规则：旧字段 → 新字段映射（或默认值）
Step 3 — 编写迁移说明（本文件对应版本段），供手动/自动迁移参考
Step 4 — 兼容性测试：用旧版 .state/ 数据在新版技能下运行，确认不报错
```

### 迁移示例（V5.2 → V6.0）
- 旧 config.json 无 `schema_version`/`domains`/`extensions` → 新版读取时补默认值
- 旧 meta.json 无 `schema_version` → 视为 v1，正常读取
- 无破坏性变更，无需人工迁移

## 三、扩展能力流程

### A. 新增功能模块
```
参考 模块注册表.md「新增模块规范」
1. 注册表登记 → 2. 创建 L3 模块文件 → 3. SKILL.md 路由登记 → 4. 知识库分片（如需要）
```

### B. 新增业务领域
```
参考 领域配置.md「新增领域流程」
1. 领域配置登记 → 2. 创建 L4 分片 → 3. index.md 登记 → 4. 时效/模板适配 → 5. config.json 启用
```

### C. 新增知识内容
```
1. 按 L4_知识库层/index.md 协议：定位分片 → Edit 追加卡片 → 同步 index
2. 写入前 grep 查重，避免重复
```

### D. 新增时效类型
```
1. 在 L2_机制层/02_时效预警机制.md 时效类型表追加行
2. 定义提前提醒天数 + 预警级别
```

## 四、演进检查清单（提交前自检）

| 检查项 | 必须 |
|--------|------|
| 单向依赖未被破坏 | ✅ |
| 无重复定义（Single Source of Truth） | ✅ |
| 旧版 .state/ 数据仍可读取 | ✅ |
| 新增输出均有可信度标注 | ✅ |
| 异常分支覆盖（信息不全/无果/不可用） | ✅ |
| 扩展点有对应规范文档 | ✅ |
| README + SKILL.md 版本号同步 | ✅ |

---

## 五、不兼容变更处理（Breaking Change）

当必须做破坏性变更（如删除层级、改 schema 结构）时：
1. 主版本号 +1（V6→V7）
2. 提供完整迁移脚本/说明
3. 在 Release Notes 明确标注"破坏性变更"及迁移步骤
4. 保留旧版数据读取的兼容窗口（至少 1 个主版本）
