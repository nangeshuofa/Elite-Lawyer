# 架构约束 — 稳定性 / 可维护性 / 可扩展性 平衡原则

> 本文件定义 Elite-Lawyer 技能的**架构铁律**。任何修改（新增模块、新增领域、升级版本）都不得违反以下约束。
> 设计目标：**适应业务变化、支持持续演进、在稳定性/可维护性/可扩展性之间取得平衡**。

## 一、分层依赖方向（单向，禁止循环）

```
L0 存储层  ← 被所有层读写（状态/配置/知识落地处）
   ↓ (提供状态与配置)
L1 内核层  → L2 机制层 → L3 模块层 → (读写) L4 知识库层
   ↑__________↑__________↑
   均不允许反向依赖或跨层越级调用
```

| 依赖关系 | 允许 | 禁止 |
|----------|------|------|
| L3 → L2 | ✅ L3 执行时引用 L2 规则 | ❌ L2 不应依赖 L3 |
| L3 → L4 | ✅ L3 读写知识库 | ❌ L4 不应调用 L3 |
| L2 → L1 | ✅ L2 引用 L1 原则/优先级 | ❌ L1 不应依赖 L2 |
| L1 → L0 | ✅ L1 读配置 | ❌ L0 不应依赖 L1 |
| L3 → L1 | ✅ 引用全局优先级 | ❌ 反过来不行 |
| 跨层（如 L3→L1 跳过 L2） | ⚠️ 仅允许"引用原则"，禁止"调用执行" | ❌ 禁止 L3 直接改 L2 规则 |

## 二、单一职责（禁止重复定义）

| 层 | 唯一职责 | 禁止 |
|----|----------|------|
| L1 | 身份、原则、全局优先级 | 定义具体业务规则 |
| L2 | 纯规则/标准/机制（流程、预警、发现、质控） | 包含具体执行步骤或业务数据 |
| L3 | 触发路由 → 执行 → 输出（引用 L2 规则） | 重复定义 L2 已有的质量标准 |
| L4 | 知识存储（被读写） | 包含执行逻辑或规则定义 |
| L0 | 状态持久化（文件读写契约） | 包含业务规则或 AI 指令 |

> **铁律**：同一条规则/标准只在一个地方定义（Single Source of Truth）。L3 需要时用"引用 L2-xx"而非复制。

## 三、扩展点（可扩展性设计）

技能通过以下**扩展点**适应业务变化，新增能力无需改动核心逻辑：

| 扩展点 | 位置 | 如何扩展 | 示例 |
|--------|------|----------|------|
| **功能模块** | `L3_模块层/` + `模块注册表.md` | 新增 `NN_模块名.md` 并在注册表登记触发词 | 新增"执行案件管理"模块 |
| **业务领域** | `领域配置.md` + `L4_知识库层/<领域>/` | 在领域配置新增领域 + 创建知识分片目录 | 新增"知识产权"领域 |
| **知识内容** | `L4_知识库层/` 分片文件 | 按 index.md 协议追加卡片 | 沉淀新裁判规则 |
| **时效类型** | `L2_机制层/02_时效预警机制.md` 时效表 | 在时效类型表追加行 | 新增"仲裁期限" |
| **模板** | `L4_知识库层/模板库/` | 新增模板卡片 | 新增"执行申请书"模板 |
| **配置项** | `.state/config.json` | 新增 JSON 字段 | 新增"常用法院列表" |
| **外部集成** | `集成接口.md` + `.state/config.json` | 在 集成接口.md 登记接口 + config 预留字段（默认关闭） | 新增"北大法宝 MCP"外部知识源 |

> 每个扩展点都有对应的"扩展规范"（见 `模块注册表.md` / `领域配置.md` / `演进指南.md`）。

## 四、禁止项（架构红线）

1. ❌ **禁止循环依赖**：任何层不得反向或跨层调用
2. ❌ **禁止重复定义**：同一规则只在一处定义（见第二节）
3. ❌ **禁止硬编码业务数据**：领域、法院、时效天数等应可配置，不写死在逻辑里
4. ❌ **禁止破坏兼容性**：升级版本时不得使旧版 `.state/` 数据失效（见 `演进指南.md` 迁移规则）
5. ❌ **禁止越级修改**：L3 不得直接修改 L2 规则文件，需走"提议→更新 L2→L3 引用"流程
6. ❌ **禁止编造兜底**：知识库无结果时必须提示，不得编造法条/案例（稳定性护栏）

## 五、三性平衡检查表

每次架构变更前，用此表自检：

| 维度 | 检查项 | 通过条件 |
|------|--------|----------|
| **稳定性** | 变更是否引入编造/错误输出风险 | 不削弱输出可信度分级与强制核验 |
| **可维护性** | 变更是否使依赖关系混乱 | 仍满足单向依赖、单一职责 |
| **可扩展性** | 变更是否堵死未来扩展 | 新增模块/领域无需改核心 |

> 三者冲突时优先级：**稳定性 > 可维护性 > 可扩展性**（法律类技能，准确性压倒一切）。
